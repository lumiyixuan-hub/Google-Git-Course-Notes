# Life Before Git - diff and patch

This section explores how software engineers manage code modifications, a discipline normally known as **version control**. At its core, version control is the “insurance policy” for any development project.

**Why is it indispensable?**
- **Traceability & Troubleshooting:** It provides a clear audit trail, letting you pinpoint exactly which change introduced a bug.
- **Seamless Collaboration:** It allows multiple developers to work on the same codebase simultaneously without overwriting each other’s progress.
- **Risk Mitigation:** It offers the ability to **roll back** to a previous stable state if a new deployment goes sideways.  

---
### **The “manual transmission” of version control: `Diff` and `Patch`**

Before modern tools like Git, developers relied on basic **Command-Line Interface (CLI)** utilities to manage changes. The two most critical tools are diff and patch.  


`Diff`: Highlighting the differences
Manually comparing files is prone to human error. Instead, we use the `diff` command to programmatically highlight the differences between two files.

**Standard Syntax:**

```
diff validations1.py validations2.py 
```
**Output:**

```
5c5,6
<	assert (type(username) == str), "username must be a string"
--
>	if type(username != str: 
> 	    raise TypeError("username must be a string"

11a13,15
>	    return False
>	# Usernames can't begin with a number
>	if username[0].isnumeric():

```

**Deciphering the Output Syntax:**

To an untrained eye, diff output looks like cryptic code. Here’s how to **decode** the normal format:
* **Action Markers:**
  * a (**Append/Add**): New content was inserted.
  * d (**Delete**): Existing content was removed.
  * c (**Change**): A specific line was modified.
* **Line Coordinates:** 5c5,6 translates to: "The 5th line in the original file was **changed** to lines 5 and 6 in the new file."
* **Directional Indicators:**
  * < refers to the **source** (original) file.
  * \> refers to the **target** (updated) file.  

---

**`diff -u` : The Unified Format**

This is the **Unified Format** (often called the "unidiff"). It is the industry standard used by Git because it’s much easier for humans to read than the "Normal Format" we discussed earlier.

```
diff -u validations1.py validations2.py 
```

**Output:**
```
@@ -2,7 +2,8 @@
 
 
 def validate_user(username, minlen):
-    assert type(username) == str, "username must be a string"
+    if type(username) != str:
+        raise TypeError("username must be a string")
     if minlen < 1:
         raise ValueError("minlen must be at least 1")
     
@@ -10,5 +11,8 @@
         return False
     if not username.isalnum():
         return False
+    # Usernames can't begin with a number
+    if username[0].isnumeric():
+        return False
     return True

```


**Breaking Down the Hunk Header:@@ -2,7 +2,8 @@**

Think of this as a **map coordinate** for your code.  

* @@ ... @@: These symbols wrap the "hunk" (a block of changes).
* -2,7 **(The Original State)**:
  * The - sign refers to the **original/source** file.
  * 2 is the **starting line number**.
  * 7 is the **number of lines** shown in this hunk from the original file (including unchanged context lines).
* +2,8 **(The New State)**:
  * The + sign refers to the **new/modified** file.
  * 2 is the **starting line number**.
  * 8 is the **number of lines** shown in this hunk for the new file.
  * *Wait, why is it 8 now?* Because you replaced 1 line with 2 lines, so the total count increased!

**The Line Markers (-, +, and Space)**
Inside the hunk, every line starts with a specific character that tells a story:
* \- **(Minus)**: This line was **removed** from the original file.
* \+ **(Plus)**: This line was **added** to the new file.
*   **(Space/Blank)**: This is a **Context Line**. It hasn't changed, but Git shows it so you know *where* you are in the code. It helps the patch command find the right spot even if other parts of the file have shifted.

---

### **The Next Step: From Identification to Implementation**
Once we have identified the discrepancies between our files, the next step is to apply those fixes to the original source code. This is where we transition from "finding" the error to "fixing" it.

**Saving the Changes**
Before we can patch a file, we must first save the output of our diff command into a permanent file (often called a "patch file") using **redirection** (>):

**Command to generate a patch file:**
```
diff validations1.py validations2.py > fix_validation.diff
```

`Patch`: Applying the differences

While diff provides the diagnosis, patch is the tool that performs the surgery. It takes the instructions generated by diff and applies them directly to the original file to bring it up to date.

**Standard Syntax:**
```
patch validations1.py < fix_validation.diff
```


**What happens next?**

By running this command, `validations1.py` is updated in place. It will now contain the exact same code as `validations2.py`. In other words, patch reads the "directions" from the `.diff` file and automatically modifies the source file for you.
